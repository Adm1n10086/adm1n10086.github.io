# shellcode执行方法

Shellcode 常见执行方法（精简好记）
本地函数指针执行把 shellcode 当字节数组，强转函数指针直接调用，适合本地测试，需关闭 DEP 或开栈可执行。
内存申请后执行调用VirtualAlloc申请 RWX 可执行内存，拷贝 shellcode 进去，跳转地址执行，绕过默认数据不可执行限制。
远程线程执行OpenProcess 拿进程句柄，VirtualAllocEx 开远程内存，WriteProcessMemory 写入 shellcode，CreateRemoteThread指向 shellcode 入口创建线程执行。
DLL 注入间接执行用远程线程调用 LoadLibraryA 注入 DLL，DllMain 里加载并运行内置 shellcode。
漏洞劫持执行栈溢出、UAF 等劫持 EIP/RIP，强行跳转到 shellcode 内存地址，靠 CPU 流水线直接执行机器码。
非代码节内存提权执行shellcode 藏在只读 / 数据节，先用 VirtualProtect 改内存为可执行权限，再跳转执行。

# kernel32基址如何获取

mov eax, [fs:30h]        指向PEB
mov eax, [eax + 0ch]  指向Ldr
mov eax, [eax + 14h]  指向InMemoryOrderModuleList
mov eax, [eax]             本进程模块
mov eax, [eax]              ntdll模块
mov eax, [eax -8h + 18h]   kernel32.Dll基址

# 缓冲区溢出、栈的溢出、堆的溢出

# 基本概念

缓冲区
程序中所使用的缓冲区可以是堆区和栈区，也可以是存放静态变量的数据区。
由于进程中各个区域都有自己的用途，根据缓冲区利用的方法和缓冲区在内存中所属区域，其可分为栈溢出和堆溢出。
缓冲区溢出漏洞
缓冲区溢出漏洞就是在向缓冲区写入数据时，由于没有做边界检查，导致写入缓冲区的数据超过预先分配的边界，从而使溢出数据覆盖在合法数据上而引起系统异常的一种现象。

# 攻击利用

将一个有效指令地址写入返回地址区域中，这样就可以让CPU跳转到我们希望执行的指令处，从而达到控制程序执行流程的目的。
攻击者通过缓冲区溢出改写的目标往往不是某一个变量，而是栈帧高地址的EBP和函数的返回地址等值。通过覆盖程序中的函数返回地址和函数指针等值，攻击者可以直接将程序跳转到其预先设定或已经注入到目标程序的代码上去执行。
在实际的漏洞利用中，由于动态链接库的装入和卸载等原因，Windows进程的函数栈帧可能发生移位，即Shellcode在内存中的地址是动态变化的，所以上述采用直接赋地址值的简单方式在以后的运行过程中会出现跳转异常。
可以在覆盖返回地址的时候用系统动态链接库中某条处于高地址且位置固定的跳转指令所在的地址进行覆盖，然后再通过这条跳转指令指向动态变化的Shellcode地址。这样便能够确保程序执行流程在目标系统运行时可以被如期进行。

SEH覆盖方法
SEH覆盖方法就是覆盖异常处理程序地址的一种攻击方式。由于SEH结构存放在栈中，因此攻击者可以利用栈溢出漏洞，设计特定的溢出数据，将SEH中异常函数的入口地址覆盖为Shellcode的起始地址或可以跳转到Shellcode的跳转指令地址，从而导致程序发生异常时，Windows异常处理机制执行的不是预设的异常处理函数，而是Shellcode。

堆溢出
从链表上卸载（unlink）一个节点的时候会发生如下操作：
node—>blink—>flink = node—>flink ;
node—>flink—>blink = node—>blink ;

通过堆溢出覆写了一个空闲堆块的块首的前向指针flink和后向指针blink，构造一个地址和一个数据，当这个空闲堆块从链表里卸下的时候，就获得一次向内存构造的任意地址写入一个任意数据的机会-Arbitrary Dword Reset（又称Dword Shoot）

# ABR例子

 #include <iostream>      // 引入输入输出库，让我们能用cout打印内容
 using namespace std;     // 用std命名空间，这样不用写std::cout，直接写cout就行
 int main() {             // 程序的入口，所有代码都要写在main里
   char * str1 = "four";  // 定义一个字符串指针，指向常量区的"four"（长度4+1个'\0'结束符，共5字节）
   char * str2 = new char[4]; // 申请一块堆内存，大小为4个char（4字节），str2指向这块内存
   char * str3 = str2;    // 让str3也指向str2的那块堆内存（两个指针指向同一块内存）
   cout << str2 << endl;  // 打印str2指向的字符串
   strcpy(str2, str1);    // 把str1的"four"复制到str2指向的内存里
   cout << str2 << endl;  // 再次打印str2
   delete str2;           // 释放str2指向的堆内存
   str2[0] += 2;          // 访问并修改已经释放的内存里的第一个字符
   delete str3;           // 再次释放同一块内存（str3和str2指向同一块）
 }

行号 注释缩写 含义 错误原因
第 5 行 //not enough space 空间不足 new char[4]只申请了 4 字节，但 "four" 实际需要 5 字节（f o u r \0），会溢出
第 7 行 //UMR 未初始化内存读取 str2刚申请，内存里是随机垃圾值，cout直接打印会读取垃圾，属于未定义行为
第 8 行 //ABW 数组越界写入 strcpy会把str1的 5 个字符（包括\0）写入只有 4 字节的str2，超出了分配的内存空间，会破坏堆结构
第 9 行 //ABR 数组越界读取 越界写入后，str2里的字符串虽然能打印，但已经破坏了相邻内存，读取时可能出现乱码或崩溃
第 11 行 //FMR and FMW 释放后读取 / 写入 第 10 行已经delete str2释放了内存，第 11 行还去修改str2[0]，属于对已释放内存的非法读写
第 12 行 //FFM 重复释放内存 str3和str2指向同一块内存，delete str3会再次释放已经释放过的内存，导致堆损坏

# 完整性

目的：为了解决未授权的修改问题，确保系统或者软件按照预期功能工作
完整性需求
所有输入的表单和参数在被软件处理前，需要根据允许输入的数据集进行比较验证，如身份征的核实，不仅能确保数据完整性，还能一定程度上防止注入
所有发行的系统功能都应该具备校验和以及散列函数的功能，以便验证该模块的准确性和完整性
所有非人类行动者都要被识别和监控，以防其对系统数据进行操作，除非有明确授权

# 可用性

目的：主要用于防范DoS攻击；不安全的编码结构，如悬挂指针，内存分配不当与无限循环结构也会影响可用性
可用性需求
在服务水平协议（Service LevelAgreement,SLA)中定义“5个9（99.999%）的软件”以确保高可用性。
确定在任何一个给定的时间点，使用该软件的用户数量的最大值。
软件和数据应该能够跨数据中心进行复制，以提供负载均衡和冗余。
确定软件的关键业务功能、关键基本功能及关键支持功能被中断后的恢复时间。

# 可认证

目的：验证并确保那些提出认证申请的实体（人、过程或者设备）的合法性和有效性
可认证需求
用户在系统中注册时需要实名认证，需要核实其手机号码等的真实性。
用户在上传、更新或删除教学资料时应当对其进行身份认证。
提供单点登录支持，并要求这些身份主体记录在预定义的用户认证列表中。
对系统核心事物的处理模块，都需要双因素或多因素身份认证。

目的：为了确认一个经过认证的实体对于请求的资源所需要的访问权限、优先级及可执行的动作而进行的权限分配方面的需求。
主/客体关系矩阵模型常用于确定授权需求。其次，要坚持最小授权原则，也就是对于主体仅授予完成工作所必需的访问权限。
下面列举一些授权需求。
高度敏感的文件必须严格限制访问，只有拥有最高许可证水平的用户才能访问
所有未经身份认证的用户（访客）将仅拥有浏览（只读）权限；而经过身份验证的用户（普通用户角色）将默认拥有浏览和评论等权限。
不同层级的教师用户拥有不同的系统权限。

# 可审计性

目的：帮助建立用户操作的历史记录，为每项活动提供一个完整的、可审计的轨迹
可审计需求

审计日志应该总是以附加（Append)的方式进行保存，而不是覆盖。

# 课本P116-119 微软软件安全开发模型

软件安全开发主要是从生命周期的角度，对安全设计原则、安全开发方法、最佳实践和安全专家经验等进行总结，通过采取各种安全活动来保证得到尽可能安全的软件。
具有代表性的4 类安全开发模型：
l) 微软的软件安全开发生命周期模型(Secure Development回ecycle, SOL), 以及相关
的敏捷SDL 和ISO/IEC 27034 标准。
2) McGraw 的内建安全模型(Building Security In, BSI), 以及BSI 成熟度模型
(Building Security In Maturity Model, BSIMM) 。
3) 美国国家标准与技术研究院(NIST) 的安全开发生命周期模型。
4) OWASP 提出的综合的轻垃级应用安全过程(Comprehensive Lightweight Application
Security Process, CLASP), 以及软件保障成熟度模型(Software Assurance Maturity Model,
SAMM) 。
以上各类模型的核心思想是，为了开发出尽可能安全的软件，把安全活动分散到软件生命周期的各个阶段中去。

# 白加黑免杀思路

# 基本概念

白加黑免杀，核心是用合法可信的 “白程序”加载恶意的 “黑文件”，借白程序的信任绕过杀软，属于分离免杀的主流思路。

# 核心原理（DLL 劫持为主）

Windows 程序加载 DLL 时，默认优先从当前目录查找，再找系统目录。白加黑利用这一规则：
找一个有签名的白 EXE（如 xxx.exe），它运行时会加载 yyy.dll；
在白 EXE 同目录放一个同名的恶意 DLL（黑 DLL）；
运行白 EXE → 系统优先加载黑 DLL → 恶意代码在白进程中执行；
杀软因信任白 EXE，放行其加载的模块，绕过查杀。

# 常见实现思路

# 经典 DLL 劫持（白 EXE + 黑 DLL）

选有签名的白 EXE，定位其要加载的 DLL 名；
编译同名恶意 DLL，放入白 EXE 目录；
运行白 EXE，触发恶意 DLL 执行。

加载应用程序的目录；
当前目录；
系统目录（使用GetsystemDirectory函数获取的路径，例如…/Windows/System32）
16位子系统的系统目录（如.…/Windows/System）；
Windows目录（使用GetwindowsDirectory获取的路径，例如.…/Windows）；
PATH环境变量里列出的目录。

# 白进程注入

启动一个合法白进程（如 explorer.exe）；
通过注入技术（如远线程、APC 注入）将恶意代码注入白进程；
恶意代码以白进程身份运行，规避检测。

# 多层白加黑（隐蔽性更强）

白 EXE → 加载 “白 + 黑” 中间 DLL（带部分合法签名）；
中间 DLL → 加载最终恶意 DLL；
层层嵌套，降低被一次性识破的概率。

# 防御要点

警惕异常目录下的系统程序（如桌面notepad.exe）；
监控白进程加载未知 DLL的行为；
启用杀软DLL 劫持防护与进程注入拦截。

# 核心原理

DLL 注入的核心原理，就是让目标进程主动加载你自己写的恶意 DLL，从而在目标进程的地址空间里执行代码。

# 核心前提

所有 Windows 进程都共用系统 DLL，因此 LoadLibrary 这类 API 的地址，在所有进程中都是相同的。
只要你有权限，就能在目标进程中写入数据、创建线程，让它调用 LoadLibrary 加载你的 DLL。

# 经典的「远程线程注入」流程

打开目标进程
用 OpenProcess 函数获取目标进程的句柄，拿到读写内存、创建线程的权限。
在目标进程中分配内存
调用 VirtualAllocEx，在目标进程的地址空间里申请一块内存，用来存放你的 DLL 文件路径。
写入 DLL 路径
用 WriteProcessMemory 把你的 DLL 完整路径字符串，写入刚才申请的内存中。
获取 LoadLibraryA 地址
用 GetProcAddress 拿到系统 LoadLibraryA 函数的地址（所有进程共用，地址不变）。
创建远程线程
调用 CreateRemoteThread，在目标进程里创建一个线程，让它直接执行 LoadLibraryA，并传入你写入的 DLL 路径。
DLL 被加载执行
目标进程加载你的 DLL，DLL 里的 DllMain 入口函数会被自动执行，你的恶意代码就在目标进程中运行了。

# 注

两个 API 作用

  1. LoadLibrary
    作用：加载 DLL 到进程内存里传入 DLL 文件名路径，就能把指定 DLL 加载进当前进程地址空间，加载成功后返回这个 DLL 的模块句柄。
  2. GetProcAddress
    作用：从已加载的 DLL 里，取出某个 API 函数的内存入口地址

# IAT HOOK原理

修改程序 IAT 表中保存的 API 函数地址，让程序调用 API 时，跳到你自己的函数执行，而不是系统原函数。

# 实验内容

IAT Hook核心思路是CrackMe.exe的IAT表中MessageBox的入口地址被修改。

IAT HOOKING是如何将函数地址由X转变为Y的，关键技术是什么
IAT Hook 把函数地址 X 改成 Y 的原理 + 关键技术
一、怎么把 X 改成 Y
程序 PE 有导入地址表 IAT，表里每一项存的都是外部 API 真实地址 X。
先解析 PE 结构，找到目标 API 在 IAT 里的表项内存地址。
用内存写入，把 IAT 表里原本存的原函数地址 X，直接覆盖成自己自定义函数地址 Y。
程序调用该 API 时，不再走原 X 地址，直接跳到 Y 地址执行自己的代码。
二、关键技术
PE 结构解析：定位 DOS 头、NT 头、导入表，精准找到目标 API 对应的 IAT 表项。
内存权限修改：IAT 所在页面默认只读，需用VirtualProtect改成可写，才能修改数据。
内存写入替换：覆盖 IAT 中原函数地址为自定义函数地址。
原函数保存：备份原始地址 X，方便钩子执行完回调原 API，不破坏原程序功能。
