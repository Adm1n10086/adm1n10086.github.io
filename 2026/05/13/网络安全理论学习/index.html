# 最常见的三种漏洞

# 越权

做到自己权限之外的事情
分为横向越权、纵向越权。
案例1
在登录账号时，我将UID改成了别人的UID，导致我登录上了别人的账号
案例2
别人填写的收货地址，我在查询我的收货地址时将地址ID改掉了，导致我查询到了别人的地址
案例3
我上传的身份证后，在查看我上传的信息时将查询该事件的id改成了别人的，导致我查询到了别人的身份证信息
案例4
我在直播间，使用用户身份抓包，修改直播间id进行了直播间禁言的操作
针对越权漏洞，我们要做的就是找到我们认为修改后可以越权的参数，例如最常见的id参数，比如userid、couponid、address.针对不同的业务要做不同的修改，比如我们现在的直播课程，如果大家打字抓包后带有roomid的参数，那么修改了roomid会怎么样。

# 并发

正常的购买逻辑

并发的购买逻辑

假如我们有1块钱，商品3个分别1块钱，由于并发并未扣除余额，三个查询余额都显示余额足够，导致可以购买，扣除玩余额之后还有-2元（有的时候会最低扣到0），相当于我们用1块钱买了3个商品
A用户像后端发起购买请求，后端收到请求后去数据库里面查询用户余额，然后后端认为用户余额够就下发商品。
但是A用户如果用并发的方式发起请求，后端同时向数据库发起多次请求，返回都成功但是还没到扣费那一步，导致了用户可以购买超出自身余额的商品

# 并发

并发并没有什么特殊场景，可以说一切场景都可以去尝试并发。

遇到sign该怎么办
遇到sign，token一些加密的东西
并发时经常会遇到带有sign的包，这时并发会导致后面的包全部并发失败（多次点击拦截多个不同的包）
并发并不局限于一个接口功能点

假如有几个参数：
1>limit
2>id

你觉得哪些参数可能有注入漏洞？
当你看到任何可控参数（可以自由修改、自定义内容，并且服务器会读取、使用这个值的请求字段），都要去观察一个注入问题,哪个都可能有注入
重点测试和数据库交互

找到注入点——再进行sqlmap，不然可能会造成一些严重后果
一个参数注入不进去的话可以尝试去从其他参数入手，后台语句语法是不一样的

场景：如果你的朋友（有一定安全基础）维护了存在以上参数的网页，但是你朋友技术不高，他对四个参数都进行了SQL注入预防，以上哪个参数可能还有SQL注入漏洞
——data，data会涉及特殊字符,如果师傅只对特殊字符处理，但是data存在特殊字符，就会报错，那么他一偷懒，可能就排列不做处理或不完善
