# Fiddler配置

勾选以下选项，其中遇见证书安装全选yes

由于默认的证书不符合android和ios的证书要求，所以，需要下载certmaker插件，双击安装后，重启fiddler。注意这一步很重要，必须使用certmaker插件，不要使用默认的证书生成器，否则抓不到包。安装插件，fiddler certmaker，百度教程
https://telerik-fiddler.s3.amazonaws.com/fiddler/addons/fiddlercertmaker.exe

# 参数

# Pirce

金额是比较常见的参数。
购买物品时经常会遇到
我们可以将金额改成-1,0.01,0.019这样的值
很少能遇到，因为金额参数最好不要前端来发送给后端,我买一瓶水，不可能是我告诉老板这瓶水多少钱，而是老板根据这瓶水来判断多少钱

# num

数量
这也是我们遇到最多的一个参数。

# id

id也是我们经常遇到的，只不过分为了很多种类，比如
product id（赠品是否可以直接购买，隐藏优惠券、会员等,可能是测试使用）
user id（越权）
address id（收货地址,下单一个别人的地址id，可能泄露别人用户地址）
为什么要修改这些id，目的是什么？
可以测试越权

# 最常见的三种漏洞

# 越权

做到自己权限之外的事情
分为横向越权、纵向越权。
案例1
在登录账号时，我将UID改成了别人的UID，导致我登录上了别人的账号
案例2
别人填写的收货地址，我在查询我的收货地址时将地址ID改掉了，导致我查询到了别人的地址
案例3
我上传的身份证后，在查看我上传的信息时将查询该事件的id改成了别人的，导致我查询到了别人的身份证信息
案例4
我在直播间，使用用户身份抓包，修改直播间id进行了直播间禁言的操作
针对越权漏洞，我们要做的就是找到我们认为修改后可以越权的参数，例如最常见的id参数，比如userid、couponid、address.针对不同的业务要做不同的修改，比如我们现在的直播课程，如果大家打字抓包后带有roomid的参数，那么修改了roomid会怎么样。

# 并发

正常的购买逻辑

并发的购买逻辑

假如我们有1块钱，商品3个分别1块钱，由于并发并未扣除余额，三个查询余额都显示余额足够，导致可以购买，扣除玩余额之后还有-2元（有的时候会最低扣到0），相当于我们用1块钱买了3个商品
A用户像后端发起购买请求，后端收到请求后去数据库里面查询用户余额，然后后端认为用户余额够就下发商品。
但是A用户如果用并发的方式发起请求，后端同时向数据库发起多次请求，返回都成功但是还没到扣费那一步，导致了用户可以购买超出自身余额的商品
